## 2026-07-15 16:58:00 -0400
Fixed the base-image bootstrap failure seen on a fresh remote host. The configured artifact checksum was sound, but the URL pointed at Ubuntu's dated daily-build tree, whose older directories are eventually removed. I changed the default to Ubuntu's retained `release-20260225` archive, which contains the same amd64 image under the release filename and the same pinned SHA256, and added a compatibility rewrite so configs generated with the former built-in URL recover automatically.

The second issue was command-plan sequencing rather than curl itself: the transfer used `check=False`, so the grouped plan continued to `mv` and `sha256sum` before Python inspected the transfer result. I made the transfer a checked command. A failed curl/copy now aborts the plan immediately, before the staging path can be promoted or hashed, and the caller receives a domain-level error explaining that the cache was not installed. I deliberately did not add a moving `current/` URL or online checksum discovery; reproducibility and a small trust surface are more important than silently following the newest image.

The remaining tradeoff is that an interrupted transfer may leave a `.part` file until the next retry. That is safe because it is never promoted, and the first command of every retry removes it. I am confident the original 404 is repaired for both new and existing configs and that future transfer failures cannot be masked by a secondary `mv: cannot stat` error.
